# AGENTS.md

Guidance for coding agents working in this repository.

## Project Overview

This is an Unreal Engine 5.7 C++ project for a Metal Gear Solid-inspired gameplay prototype. The Unreal project file is `MGS.uproject`; the primary runtime module is `MGS`.

Use `SPEC.md` as the gameplay design reference. When implementing stealth, alert, combat, HUD, input, item, or encounter behavior, check the relevant section of `SPEC.md` before inventing behavior.

## Repository Layout

- `Source/MGS/` contains the game C++ module.
- `Source/MGS.Target.cs` builds the standalone game target.
- `Source/MGSEditor.Target.cs` builds the editor target.
- `Config/` contains checked-in Unreal project settings.
- `Content/` contains Unreal assets and Blueprint content.
- `Binaries/`, `Intermediate/`, `Saved/`, and `DerivedDataCache/` are generated by Unreal and should not be edited or committed.

## Coding Guidelines

- Follow existing Unreal C++ conventions in this project.
- Prefer explicit includes and keep public headers minimal.
- Use Unreal reflection macros (`UCLASS`, `USTRUCT`, `UENUM`, `UPROPERTY`, `UFUNCTION`) only where Blueprint/editor exposure, serialization, GC tracking, networking, or reflection is needed.
- Keep gameplay behavior in C++ when it defines reusable systems or rules. Use Blueprints and assets for tuning, presentation, level setup, animation hookups, and editor-authored references.
- Use the existing `LogMGS` category for project logging.
- Avoid hard references to assets in C++ unless there is an established local pattern or a strong reason.
- Do not make broad refactors while implementing a small gameplay request.

- **Unreal Engine C++ Best Practices and Patterns**
- **KISS** — simplest thing that works
- **YAGNI** — don't build for hypothetical needs
- **DRY** — remove real duplication, not shape-similar code
- **Locality of change** — adding a new entity or feature should touch as few files as possible


## Unreal-Specific Notes

- The module currently depends on `Core`, `CoreUObject`, `Engine`, `InputCore`, `EnhancedInput`, `AIModule`, `StateTreeModule`, `GameplayStateTreeModule`, `UMG`, and `Slate`.
- The project uses Enhanced Input. Preserve that path for player input unless the request explicitly changes it.
- The `.uproject` enables StateTree, GameplayStateTree, and the Modeling Tools editor mode plugin.
- Generated project files such as `.sln` can be recreated by Unreal; do not hand-edit them as source of truth.

## Build And Validation

From the repository root, build with UnrealBuildTool when Unreal Engine 5.7 is available locally. Typical commands on Windows are:

```powershell
& "C:\Program Files\Epic Games\UE_5.7\Engine\Build\BatchFiles\Build.bat" MGSEditor Win64 Development -Project="$PWD\MGS.uproject" -WaitMutex
& "C:\Program Files\Epic Games\UE_5.7\Engine\Build\BatchFiles\Build.bat" MGS Win64 Development -Project="$PWD\MGS.uproject" -WaitMutex
```

If the engine is installed elsewhere, locate the matching UE 5.7 install and use its `Build.bat`.

For editor smoke checks, launch the project in the Unreal Editor and verify that maps/assets affected by your change load without Blueprint compile errors. If you add gameplay systems, prefer adding a focused automated test or a small reproducible validation path.

## Asset And Config Changes

- Treat binary Unreal assets (`.uasset`, `.umap`) carefully. Make sure changes are intentional and tied to the request.
- Keep `Config/Default*.ini` edits focused. Many editor operations can rewrite unrelated settings.
- Do not delete or regenerate asset folders unless the user asks for that explicitly.

## Git Hygiene

- The worktree may contain user changes. Do not revert changes you did not make.
- Keep generated folders out of commits: `Binaries/`, `Intermediate/`, `Saved/`, `DerivedDataCache/`, and local IDE metadata.
- Before finishing a coding task, summarize changed files and any validation performed.
